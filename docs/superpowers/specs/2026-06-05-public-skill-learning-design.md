# 公共技能学习设计

## 背景

AgentMa 已经有 Skills 页面、模拟技能数据，以及一个把 workspace 技能复制到用户技能目录的后端安装链路。接下来要做 Skills 市场，需要先把“我的技能背包”和“公共技能”的产品语义收敛清楚。

已确认的规则是：公共技能不支持运行时引用。用户只能点击“学习技能”，把公共技能复制到“我的技能背包”。学习完成后，这个副本就是用户自己的技能，可以编辑，也可以二创。

## 目标

- 把技能分成两个产品空间：“我的技能背包”和“公共技能”。
- 把“学习技能”作为加入我的技能背包的第四种方式。
- 让所有可运行、可编辑的技能都落在我的技能背包里。
- 防止公共技能更新影响已经学习过的用户副本。
- 允许用户和 agent 只对背包技能进行二创，发布公共技能必须是单独的显式动作。

## 非目标

- 不支持运行时引用公共技能。
- 不支持公共技能到已学习副本的自动同步。
- 第一版不做公共技能更新后的合并或 rebase。
- 第一版不做版本选择 UI。

## 产品模型

### 我的技能背包

我的技能背包是用户自己的可编辑技能集合。这里的每个技能都可以启用、停用、编辑或二创。用户要求创建技能、修改技能或在 agent 对话过程中二创技能时，目标都应该是背包里的技能。

背包技能可以来自四种方式：

1. 内置或已有的本地用户技能。
2. 从 GitHub 导入。
3. 从 Workspace 抽取。
4. 从公共技能学习。

### 公共技能

公共技能是可发现、可学习的模板。公共技能不会直接挂载进 agent 会话，也不会被背包技能引用。

公共技能可以由作者或有权限的发布者更新，但更新只改变公共空间里的技能，以及之后新学习的人拿到的副本。已经学习到背包里的副本不受影响。

### 学习公共技能

点击“学习技能”会把选中的公共技能包复制进当前用户的技能背包。复制后的技能：

- `location` 为 `"user"`；
- 立即可编辑；
- 可以像其他背包技能一样启用或停用；
- 可以保存来源归因信息，用于展示和审计；
- 不订阅公共技能的未来更新。

如果用户背包里已经有同名的标准化技能名，学习动作返回冲突。UI 应提示用户选择一个新名称后再重试。

## 数据形态

公共技能目录记录应该和已安装技能记录分开。

```ts
interface PublicSkill {
  id: string;
  slug: string;
  name: string;
  description: string;
  authorSub: string;
  revision: number;
  bundlePath: string;
  publishedAt: number;
  updatedAt: number;
}

interface LearnedSkillMetadata {
  learnedFromPublicSkillId?: string;
  learnedFromPublicRevision?: number;
  learnedAt?: number;
}
```

`SkillInfo` 可以保留当前运行时形态，从公共技能学习而来的副本只需要额外保存可选来源元数据。

## API 设计

第一版应该增加后端 API，不继续扩展当前只靠 localStorage 的 Skills 状态。

- `GET /api/skills/public`：列出公共技能。
- `GET /api/skills/public/:id`：读取公共技能详情。
- `POST /api/skills/public/:id/learn`：把公共技能复制到当前用户的技能背包。
- `POST /api/skills/public`：把背包技能发布到公共空间。
- `PATCH /api/skills/public/:id`：更新公共技能信息或技能包。

学习接口应该复用 workspace 安装的安全复制规则：必须存在 `SKILL.md`，校验 frontmatter，禁止符号链接，限制文件数量和总大小，先复制到临时目录，再原子重命名到最终用户技能路径。

## UI 设计

Skills 页面应该提供两个一级视图：

- “我的技能背包”：展示已安装技能、编辑入口、启用/停用控制，以及来源归因。
- “公共技能”：展示可搜索的公共目录，主要操作是“学习技能”。

公共技能列表不应该出现“启用”或“使用”，因为公共技能不能直接挂载。主按钮只应该是“学习技能”。

从公共技能学习来的背包技能，可以展示“来源：公共技能 <name>”一类的信息。这个信息只用于归因，不代表同步关系。

## 发布规则

发布会把一个背包技能复制到公共空间。更新公共技能只改变公共空间里的记录和技能包，绝不写入其他用户的技能背包。

虽然公共更新不会影响已经学习的副本，但会影响未来学习的人。因此发布和更新动作需要明确确认，确认内容包括技能名、revision 变化和简短描述。

Agent 驱动的编辑应该只作用于背包技能。如果用户想把编辑结果发布出去，发布必须是一个单独的显式动作。

## 错误处理

- 公共技能不存在或不可用：返回 `404`。
- 背包技能名冲突：返回 `409`，并带上已存在的名称。
- `SKILL.md` frontmatter 无效：返回 `400`。
- 技能包包含不安全内容、符号链接或超过大小限制：返回 `400`。
- 无权发布或更新公共技能：返回 `403`。
- 复制中途失败：清理临时目录，保持用户背包不变。

## 测试

- 单元测试公共学习的复制隔离：学习后更新公共技能，确认已学习副本不变。
- 单元测试同名技能冲突。
- 单元测试安全复制校验，包括符号链接、超大目录和缺失 frontmatter。
- API smoke：发布技能、列出公共技能、学习技能、确认它作为用户技能出现。
- UI smoke：公共目录学习流程、同名错误、背包技能可编辑入口。

## 实现备注

当前 `Skills.tsx` 的状态主要由 localStorage 支撑，而 workspace 安装已经有后端复制 API。公共技能学习功能应该把技能发现和已学习状态逐步迁到后端 API，以保证公共目录、审计日志和租户边界可靠。

第一版应该保留现有 workspace 安装行为，在旁边新增公共技能学习，不尝试解决公共更新合并、版本对比或自动同步。
